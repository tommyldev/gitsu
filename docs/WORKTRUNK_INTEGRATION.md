# gitsu → worktrunk integration map

The single source of truth for **UI affordance → `wt` invocation**.
Every gitsu feature that touches worktree state goes through here.
If a worktrunk subcommand changes its flags, output schema, or
defaults, this file is the first place to update.

> **Rules**
> 1. Always pass `--format=json` for anything we parse.
> 2. Never call the interactive picker (`wt switch` with no args).
> 3. Every `wt` call runs in the target worktree's CWD.
> 4. TTY is off by default; reserve for the per-worktree terminal.

## Lifecycle

| UI action                              | `wt` invocation                                          | IPC command              |
|----------------------------------------|----------------------------------------------------------|--------------------------|
| Open repo                              | — (validate `.git`)                                      | `open_repo`              |
| Open repo, discover defaults           | `wt config state default-branch`                          | `wt_config_state_default_branch` |
| List worktrees                         | `wt list --format=json`                                  | `wt_list`                |
| Create worktree from default base      | `wt switch --create <branch> --format=json`              | `wt_switch_create`       |
| Create worktree from a specific base   | `wt switch --create <branch> --base <base> --format=json`| `wt_switch_create`       |
| Create + run command (editor/terminal) | `… --execute '<cmd>'`                                    | `wt_switch_create`       |
| Switch to existing worktree            | `wt switch <branch> --format=json`                       | `wt_switch`              |
| Remove worktree                        | `wt remove <branch> --format=json`                       | `wt_remove`              |
| Remove + delete branch                 | `… --delete-branch`                                      | `wt_remove`              |
| Force remove (dirty)                   | `… --force`                                              | `wt_remove`              |
| Merge into default                     | `wt merge <default> --format=json`                       | `wt_merge`               |
| Merge, skip hooks                      | `… --no-hooks`                                           | `wt_merge`               |

## Recurring operations

| UI action                              | `wt` invocation                                          | IPC command              |
|----------------------------------------|----------------------------------------------------------|--------------------------|
| Commit with LLM message                | `wt step commit --format=json`                           | `wt_step_commit`         |
| Commit with options (stage mode)       | `… --stage=tracked|all|none`                             | `wt_step_commit`         |
| Commit dry-run (preview message)      | `… --dry-run`                                            | `wt_step_commit`         |
| Copy `.env` + caches to a new wt       | `wt step copy-ignored --from=<base> --to=<new> --format=json` | `wt_step_copy_ignored` |
| Recopy ignored files (force)           | `… --force`                                              | `wt_step_copy_ignored`   |
| List hooks                             | `wt hook show --format=json`                             | `wt_hook_show`           |
| Manually run a hook                    | `wt hook <type> [-- <args>]`                             | (planned: M6)            |
| Approve a hook command                 | `wt config approvals add`                                | (planned: M6)            |

## Hook installation (the gitsu → worktrunk bootstrap)

When a user opens a repo for the first time, gitsu offers to install
a default `post-start` hook in `.config/wt.toml` that does
`wt step copy-ignored` — the ergonomic glue that brings `.env`,
`node_modules/`, `target/`, etc. into new worktrees.

**File written:** `<repo>/.config/wt.toml`

```toml
# Generated by gitsu. Edit / remove in the Hooks manager.
# Docs: https://worktrunk.dev/hook/

[post-start]
copy = "wt step copy-ignored"
```

If the file already exists, gitsu only adds a `[post-start]` section
if not present (never clobbers user hooks).

## Switch & directory change

**The tricky problem:** Tauri webviews don't have a CWD the way
terminals do. gitsu can't `cd` the user's shell from a button.

**The solution:** gitsu treats every worktree as a path-addressable
unit. All git operations use absolute paths to the worktree's
directory; the per-worktree terminal (M5) is `cwd`'d to the worktree
at spawn time. The user runs `wt switch` from their shell if they
want the terminal session's CWD to follow worktrees — gitsu is
agnostic to that workflow and never assumes it.

## Approvals

`wt` prompts for approval on first run of project-defined hook
commands. The prompt is structured, so gitsu captures stderr and
parses it into an `ApprovalRequest`:

```
▲ repo needs approval to execute 3 commands:

○ pre-start install:
  npm ci
○ pre-start build:
  cargo build --release
○ pre-start env:
  echo 'PORT={{ branch | hash_port }}' > .env.local

❯ Allow and remember? [y/N]
```

Surfaced in the UI as a modal with per-command checkboxes + an
"Allow and remember" button. Backend re-spawns the `wt` command
with `--yes` after the user confirms.

## Version pinning

`gitsu` ships a specific `wt` version. The build pipeline:

1. `scripts/download-wt.sh` reads `WT_VERSION` (default `0.56.0`).
2. Downloads the matching `worktrunk-<triple>.<ext>` release.
3. Verifies SHA-256 against the upstream `*.sha256` file.
4. Stages it as `src-tauri/binaries/wt-<triple>` (Tauri sidecar).
5. `tauri.conf.json` declares it via `bundle.externalBin`.

The frontend checks `wt_version` on every repo open and warns if the
sidecar is missing or below `MIN_WT_VERSION` (currently `0.55.0`).

## Capabilities (Tauri 2 permissions)

The `wt` sidecar is declared in `src-tauri/capabilities/default.json`
with both `shell:allow-execute` and `shell:allow-spawn` permissions,
scoped to `name: "wt", sidecar: true`. Other shell access is denied.

## Things we explicitly do NOT do

- We don't call `wt switch` with no arguments (the interactive picker).
- We don't parse the colored `wt list` text output — only JSON.
- We don't re-implement hook installation logic. We just write the
  TOML and let `wt` validate.
- We don't try to handle worktrunk's `wt config` subcommand beyond
  the small set we use. If a user wants more, they can use `wt`
  directly in their terminal.
- We don't fork `wt`. If a feature feels like it should live in
  `wt`, we propose it upstream.
